# 已知限制和下一阶段

这个 POS 当前已经可以作为云端多分行测试版使用，但正式大量使用前还有一些需要继续加强的地方。

## 库存并发

当前做法：

- 收款时先扣本机库存。
- 在线时优先使用 Firestore Transaction 写入销售并扣减云端库存。
- 在线 transaction 失败时，本机订单和库存会回滚。
- 断网时把库存变更加入待同步队列。

限制：

- 在线交易已经降低库存覆盖风险，但离线期间多设备销售同一库存，恢复网络后仍可能需要人工复核。

下一阶段建议：

- 使用 Cloud Functions 作为唯一收款入口，进一步集中校验权限、库存和订单写入。
- 增加“库存冲突复核”后台页面。

## 离线冲突

当前做法：

- 断网订单先存本机。
- 恢复网络后自动补传订单和库存。

限制：

- 如果离线期间多台设备卖出同一商品，恢复网络后需要人工核对库存。

下一阶段建议：

- 增加库存调整记录。
- 增加冲突提示。
- 管理员后台显示“需要复核”的订单。

## 权限

当前做法：

- Firebase Google 登录。
- Firestore Rules 限制管理员和授权员工。
- 本机仍保留离线缓存和本地授权资料。

限制：

- 本机离线资料适合断网继续工作，但不是服务器级权限来源。

下一阶段建议：

- 登录后定期刷新云端授权状态。
- 对被停用用户，恢复网络后立即锁定 POS。

## 部署配置

当前做法：

- 使用 `firebase-config.local.js` 保存真实 Firebase 配置。
- 该文件不提交 GitHub。

注意：

- 上线部署时必须确保部署版本能读取真实 Firebase 配置。
- API Key 应限制 HTTP referrers。
